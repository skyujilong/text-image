小说转多媒体系统 — 整体架构概览

基于 LangChain + LangGraph 的自动化流水线，请确认这是否符合你的设想

【入口】 指定目录
📄 config.json   ← 角色信息 / 世界观 / 题材
📁 chapters/     ← 各章节小说原文
📁 summaries/   ← 各章节摘要
↓
Phase 1 — 角色初始化
解析 JSON → 角色音色抽卡（本地TTS试音）→ 角色图片抽卡 → ComfyUI生成角色参考图
⚠ 固定角色特征参数（发色/体型/服装 LoRA等），防止人物崩坏
产物保存至  output/{小说名}/characters/
↓ 循环每一章节
Phase 2 — 章节改编
原文 → LLM改编为多角色口播剧本 → LLM自审 → 人工审核
剧本 → LLM生成分镜稿（场景/情绪/构图）→ LLM自审 → 人工审核
⚠ 不通过 → 回退上一步重新生成
↓
Phase 3 — 音频合成
剧本台词 → 远端TTS服务器 大规模合成（按角色音色）
输出：各段落音频文件 + 时间轴（每段起止时间戳）
产物保存至  output/{小说名}/{章节}/audio/
↓
Phase 4 — 场景图生成
分镜稿 × 音频时间轴 → ComfyUI批量生成场景图
⚠ 图片时序 = 音频骨架，图片是皮（按音频段落切换）
产物保存至  output/{小说名}/{章节}/images/
↓
人工决策节点 — 章节结束
询问：是否将已积累的 N 章内容导入剪映？
✅ 是 → 打包导出剪映工程 → 标记章节状态为「已提交」
❌ 否 → 继续下一章节循环
↓
Phase 5 — 剪映导出
音频轨 + 图片序列 + 时间轴 → 生成剪映草稿工程文件
状态记录：chapters_status.json 跟踪哪些章节已提交

👆 以上是根据你的描述梳理出的整体流程。请在终端告诉我：这个流程是否准确？有没有遗漏或不对的地方？
